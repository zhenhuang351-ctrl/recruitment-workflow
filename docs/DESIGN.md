# Recruitment Workflow 设计说明

## 要解决的问题

复杂岗位的招聘问题常常起于模糊的需求：什么算“有 AI 经验”、哪些背景只是加分、哪些条件必须核实、如何避免关键词相似却项目深度不足。若这些规则未被写清，自动化只会放大误判。

本项目提供一套本地、可迁移的工作流：先澄清岗位标准，再用证据评估材料，随后把电话、面试反馈和复盘写回可追溯的事实源。

## 核心立场

AI 整理招聘证据，招聘者承担判断与决策。

| AI 可协助 | 必须人工确认 |
| --- | --- |
| 提取材料事实、归类证据、提示缺口、起草问题与纪要 | 确认岗位标准、解释灰度信息、联系候选人、约面、终止、录用、变更规则 |

材料没有明确写到的内容一律标记为“待验证”，不由模型补写。分数表示材料中已出现的证据强度，不是对候选人的结论。

## 文件分层与长期记忆

每个岗位独立保存以下内容：

1. `CONTEXT.md`：确认过的规则、待确认问题和下一步；
2. `ROLE_STANDARD.md`：版本化岗位标准、资格口径、能力维度和验证问题；
3. `PIPELINE.json`：招聘者确认的主阶段名称、顺序和阶段状态；
4. `SOURCING_STRATEGY.md` 与 `KEYWORD_ITERATIONS.md`：搜寻假设及其真实命中证据；
5. `candidate-ledger.xlsx`：候选人流程的唯一事实源；
6. 候选人 Markdown 档案：可追溯的简历、电话和面试证据；
7. `FEEDBACK_ITERATIONS.md`：人工反馈、适用范围和待确认的改进建议。

HTML 看板只从 Excel 台账生成，不反向修改台账。岗位、候选人和沟通资料均不放入本仓库；仓库只提供模板、Skill 和本地工具。

## 年龄、年限、隐私与偏见

年龄、学历和工作年限并非默认规则。岗位澄清必须明确：它是硬性标准、评分参考还是电话核实项；没有人工确认就不能参与判断。即使被设为硬性标准，AI 仍只给出待人工确认提示，不自动终止流程。

工作年限要区分总工作年限、对口岗位年限和对口项目年限。候选人的个人信息遵循最小化原则：只处理有权限使用的材料，避免把真实资料上传公开仓库或无关服务。

## 反馈校准

反馈先分为五层：流程事实、能力反馈、岗位匹配判断、岗位级改进建议和待澄清问题。反馈的适用岗位或招聘环节不清楚时，先确认“这条反馈具体适用于哪个岗位或招聘环节？”，再决定是否沉淀为标准或策略调整。

少于五条同类、范围已确认的反馈时，系统只记录观察。达到门槛后，才可建议补充验证问题、细化维度、调整建议年限或更新关键词。所有变更均需人工确认、生成新版本并保留差异记录；模型不能自行学习或静默调权。

## 明确不做的事情

- 不登录、抓取或绕过招聘平台；
- 不自动发消息、约面、拒绝、淘汰或录用；
- 不处理验证码、风控、付费或平台限制；
- 不用黑箱匹配分掩盖证据缺口；
- 不把真实招聘数据放入公开仓库。

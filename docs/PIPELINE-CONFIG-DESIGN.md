# Pipeline 配置设计

## 目标

不同岗位、不同公司的招聘流程名称和顺序不同。每个岗位目录保存独立的 `PIPELINE.json`，由招聘者确认本岗位的主阶段名称和顺序。Excel 台账和正式招聘复盘看板的漏斗读取同一份配置。

## 配置文件

每个岗位目录创建 `PIPELINE.json`：

```json
{
  "stages": [
    { "id": 0, "name": "简历初筛" },
    { "id": 1, "name": "电话沟通" }
  ],
  "statuses": ["待处理", "待招聘者确认", "待候选人回复", "已约面", "进行中", "已完成", "暂缓", "终止"]
}
```

- `id` 决定漏斗与 Excel 下拉的顺序，必须从 `0` 开始连续编号；`name` 由招聘者在岗位澄清中确认。
- Excel 主阶段显示为 `id-name`，例如 `0-简历初筛`。
- 阶段状态独立保存。候选人暂缓或终止时，主阶段保留实际停留节点，状态写“暂缓”或“终止”。

## 交互与数据流

1. 新建岗位时，AI 先询问“请按实际顺序列出该岗位的招聘阶段名称”。
2. 招聘者确认后，AI 写入 `PIPELINE.json`，并在 `ROLE_STANDARD.md` 留下配置版本与确认记录。
3. 台账生成脚本读取该文件，写入主阶段和阶段状态下拉。
4. 正式看板同步脚本读取 Excel 的同一份阶段配置；漏斗严格按 `id` 排序，即使某些阶段当前人数为零也保留显示。
5. 未提供配置时，脚本不擅自猜测流程，提示先完成岗位澄清。

## 边界与验证

- 不自动修改已确认流程；阶段改名、插入、删除或调整顺序须人工确认并留版本记录。
- 配置必须包含至少一个阶段；阶段 `id` 不可重复，阶段名称不可为空。
- 测试将覆盖：自定义阶段写入台账、漏斗按自定义顺序输出、暂缓/终止仍仅属于阶段状态。

# 招聘证据工作流路由器

这是一个可被 Codex、ChatGPT、Claude、Cursor 等常见 AI 工具读取的本地招聘工作流。开始任何候选人操作前，先读取：

1. `00-招聘规则.md`
2. 对应岗位目录中的 `CONTEXT.md`（若已存在）
3. 对应岗位目录中的 `ROLE_STANDARD.md`
4. 对应岗位目录中的 `PIPELINE.json`
5. 对应岗位目录中的 `candidate-ledger.xlsx`

Excel 是候选人流程数据的唯一事实来源；候选人 Markdown 档案用于保存可追溯证据；岗位目录中的 `招聘数据复盘.html` 由正式招聘复盘看板模板创建，并从台账同步招聘漏斗、来源、流失原因与时效数据。台账主阶段统一保存为 `编号-名称`。

## 识别意图并执行

| 用户意图 | 必读文件 | 交付物 | 写回范围 |
| --- | --- | --- | --- |
| “帮我澄清/新建一个岗位” | `00-招聘规则.md`、`templates/岗位澄清提示词.md` | `ROLE_STANDARD.md`、`SOURCING_STRATEGY.md`、`PIPELINE.json`、`CONTEXT.md`、`KEYWORD_ITERATIONS.md` | 新建岗位目录，等待招聘者确认标准与流程 |
| “更新搜寻策略/做人才盘点” | 岗位上下文、岗位标准、`SOURCING_STRATEGY.md`、`KEYWORD_ITERATIONS.md` | 对标公司、Title、项目、技能词、检索式、命中证据与拟议变化的结构化草稿 | 只记录观察；生效变化须人工确认 |
| “评估这份简历” | 岗位标准、台账、`templates/简历评估提示词.md` | 结构化简历、证据卡、电话问题 | 新建或更新候选人档案；新增前先去重，并确认简历来源与收取时间 |
| “这是电话沟通情况” | 岗位标准、候选人档案、台账、`templates/电话沟通提示词.md` | 事实型电话纪要、下一步建议 | 仅在招聘者确认后更新阶段、终止原因或约面信息 |
| “这是面试官反馈” | 岗位标准、候选人档案、台账、`templates/面试反馈提示词.md` | 反馈摘要、补证项、策略迭代建议 | 记录候选人进展；策略变更写入待确认日志 |
| “这是 Offer / 入职进展” | 岗位标准、`PIPELINE.json`、候选人档案、台账、`templates/Offer与入职提示词.md` | 主阶段、阶段状态、终止原因、备注和下一步动作草稿 | 人工确认后更新台账中的主阶段、阶段状态、终止原因、备注和下一步动作 |
| “更新复盘/看招聘数据” | 台账、`PIPELINE.json`、`招聘数据复盘.html` | 正式看板中的漏斗、时效、来源、终止原因及待办 | 从 Excel 同步数据与阶段顺序；不反向覆盖台账 |

## 不可替代的人工判断

- AI 只整理材料和提示补证，不自动发送消息、抓取招聘平台、约面、拒绝、淘汰、录用或修改岗位标准。
- 所有改变候选人阶段、终止流程、约面安排和岗位规则版本的动作，均须人工确认后写回。
- 未出现在材料或沟通记录中的信息一律标记为“待验证”，不能编造、脑补或当作负面证据。
- 年龄和工作年限只能按照当前 `ROLE_STANDARD.md` 中已经确认的规则处理；没有配置，不得自行使用。

## 模糊反馈拦截

- 反馈应区分流程事实、能力反馈、岗位匹配判断、岗位级改进建议和待澄清问题，避免把个人流程结论误当成通用规则。
- 适用范围不明确时，一次只问一个澄清问题：**这条反馈具体适用于哪个岗位或招聘环节？**
- 未澄清前，可以原样记录反馈并标记“待澄清”，但不得修改岗位标准、评分或搜寻策略。

## 每次输出的基本格式

简历评估先输出一行“姓名｜建议｜能力证据得分｜证据覆盖率”，再给不超过三条整体判断，随后才展开结构化信息与证据细节。所有未确认信息统一以加粗的“**待核实**”呈现。每一条能力判断都要保留原文摘录或明确来源位置。
